# NEXIO 개발 절대규칙

이 문서는 `NEXIO`에서 예외 없이 지켜야 하는 개발 기준이다.

## 1. 작업 범위

- 사용자가 지정한 범위만 수정한다.
- 요청 범위를 넘는 임의 개선, 임의 정리, 임의 리디자인을 하지 않는다.
- 기능 추가, 버그 수정, 리팩터링, 배포 준비를 한 번에 섞지 않는다.
- 값을 추정해서 넣지 않는다. 계좌, 금액, 날짜, 전략값, 운영 기준은 확인된 값만 반영한다.
- 확실하지 않은 정보는 임의로 쓰지 않는다.
- 기존 사용자 변경을 되돌리거나 덮어쓰지 않는다.
- 배포와 무관한 변경을 끼워 넣지 않는다.

## 2. 구조 및 구현

- 500줄 초과 파일은 분리 대상이다.
- 큰 파일을 수정할 때는 분리 가능성을 먼저 본다.
- 한 번에 여러 대형 파일을 동시에 키우지 않는다.
- 단일 책임을 지킨다.
- UI, 엔진 로직, DB 헬퍼, 외부 API 연동, 알림 로직을 섞지 않는다.
- 공통 의미의 값과 라벨은 여러 곳에 중복 하드코딩하지 않는다.
- 표시용 라벨과 실행용 로직을 분리한다.
- 수동 개입 경로도 자동 엔진과 같은 안전장치를 타게 유지한다.
- 브로커 상태와 DB 상태가 어긋날 가능성을 항상 전제로 구현한다.

## 3. 엔진 및 매매 안전 규칙

- 실전 규칙은 UI보다 우선한다.
- 손절, 예산, 엔진 락, 정합성, 주문 실패 누적 같은 안전 규칙은 임의로 약화하지 않는다.
- 매매 로직은 추측으로 바꾸지 않고 코드에 명시된 조건으로만 바꾼다.
- 손실, 청산, 재진입, 포지션 제한 규칙은 한 군데에서 일관되게 관리한다.
- 운영 중단 조건이 있으면 신규 진입보다 중단이 우선한다.

## 4. 서버 검증 및 보안

- 모든 API route는 입력값을 서버에서 검증한다.
- 인증, 권한, secret 헤더, 세션 검증을 클라이언트 판단에 맡기지 않는다.
- 토큰, API 키, 계좌정보, 비밀값을 코드에 하드코딩하지 않는다.
- 민감정보는 응답, 로그, 알림에 불필요하게 노출하지 않는다.
- 브로커 주문/계좌 API는 실패 시 원문 맥락을 남기되, 비밀정보는 제외한다.

## 5. 성능 및 데이터 접근

- N+1 조회를 만들지 않는다.
- Supabase `select`는 필요한 컬럼만 지정한다.
- 반복 조회는 배치 처리와 캐시 가능한 구조를 우선한다.
- 엔진 경로에서 불필요한 네트워크 호출과 중복 조회를 늘리지 않는다.

## 6. 폴더 구조

- `app/api/`는 라우트 처리만 담당한다.
- `components/`는 UI만 담당한다.
- `hooks/`는 상태와 화면용 비동기 흐름만 담당한다.
- `lib/engine/`는 엔진 비즈니스 로직만 담당한다.
- `lib/kis/`는 브로커 연동과 프로필/토큰 관리만 담당한다.
- 새 파일도 같은 역할 체계를 따른다.

## 7. 검증 규칙

- 코드 수정 후 최소 `./node_modules/.bin/tsc --noEmit --noUnusedLocals --noUnusedParameters`를 확인한다.
- 엔진/상태/도메인 로직 수정 후 `npm run test:unit`을 확인한다.
- 배포 전 `npm run lint`, `npm run build`, `npm run check:engine`를 우선 확인한다.
- 실패한 검증을 무시하고 다음 단계로 넘어가지 않는다.
- 장중 장애나 정합성 이슈는 증상만 가리지 말고 원인을 확인한다.

## 8. 배포 규칙

- 커밋되지 않은 변경이 있으면 배포하지 않는다.
- 배포와 무관한 변경이 섞여 있으면 정리 후 배포한다.
- 검증 없이 바로 운영 배포하지 않는다.
- 운영 URL 반영 후 최소 응답 상태와 핵심 엔드포인트를 확인한다.
- 실패한 배포를 덮어두고 다음 작업으로 넘어가지 않는다.

## 9. 세션 시작 규칙

- 세션 시작 시 이 문서와 `NEXT_SESSION.md`, `docs/operations/pre-live-checklist.md`를 먼저 확인한다.
- 우선순위 충돌 시 전략 개선보다 운영 안정성, 정합성, 주문 안전성, 장애 복구를 우선한다.
