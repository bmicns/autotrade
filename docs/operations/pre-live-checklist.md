# Pre-Live Checklist

실제 자금 투입 전 `NEXIO` 운영 체크리스트.

이 문서는 "운영 전에 반드시 끝낼 것"만 모은 기준 문서다.
전략 개선보다 운영 안정성, 정합성, 주문 안전성, 장애 복구력을 먼저 점검한다.

## 0. 사용 원칙

- `pass` / 허용 가능한 `warn`만 남은 상태에서만 운영 시작
- `fail` 항목이 1개라도 있으면 실운용 금지
- `warn` 항목은 "거래 가능 경고"인지 "운영 중단 경고"인지 별도 분류
- `preflight.readiness.autoTradingReady=true` 인지 확인
- `preflight.readiness.livePromotionReady=true` 가 아니면 실전 승격 보류
- 운영 전 최종 판단은 `preflight`, `engine-log`, 실계좌 상태, 브로커-DB 정합성을 함께 본다

## 1. 빌드 / 정적 검증

- `npm run build` 통과
- `./node_modules/.bin/tsc --noEmit --noUnusedLocals --noUnusedParameters` 통과
- `npm run test:unit` 통과
- `npm run lint` 통과
- `npm run check:engine` 통과
- 로컬 점검 스크립트가 실제 라우트 호출 형태와 맞는지 확인

## 2. 배포 / 환경 분리

- 실전과 모의 환경이 명확히 분리되어 있는지 확인
- 실전/모의 계좌, 토큰, 알림, DB 기준을 혼동하지 않도록 표시 확인
- 로컬 `.env`, 배포 환경변수, DB 설정 간 drift 없음 확인
- 장중 배포 금지 원칙 확인
- 롤백 절차 문서 존재 확인

## 3. 인증 / 권한 / 관리자 보호

- `/api/engine`, `/api/daily-report`, `/api/market-close`, `/api/observer`, `/api/learn` 크론 보호 확인
- 로그인 없이 관리자 API 접근 시 `401` 또는 적절한 차단 확인
- 세션 로그인 후 수동매수/수동매도/설정 변경 가능 확인
- 쓰기 API의 origin / referer 보호 동작 확인
- 운영 액션이 감사 로그 또는 이벤트 로그에 남는지 확인

## 4. KIS 설정 / 토큰 / 계좌

- `kis_config`에 운용할 계정 정보 저장 확인
- `app_config`의 계좌 관련 설정과 실제 계좌 일치 확인
- 활성 프로필이 의도한 계좌를 참조하는지 확인
- 토큰 재발급 단독 테스트 성공 확인
- 저장 토큰 만료 경고가 "앱키 만료"로 오해되지 않도록 문구 확인
- 저장 토큰이 없어도 엔진 실행 시 신규 발급으로 정상 작동하는지 확인
- 토큰 재발급 rate limit 오류 시 경고/정지/알림 동작 확인
- 계좌번호 불일치 오류 발생 시 즉시 차단되는지 확인
- 잔고 조회, 시세 조회, 주문 API 각각 단독 확인
- 주문 계좌 인증 확인 API가 `pass` 상태인지 확인

## 5. 엔진 제어 / 실행 조건

- `engine_enabled=true` 상태 확인
- `engine_lock` 동작 확인
- stale lock 회복 동작 확인
- 중복 cron 실행 방지 확인
- `max_positions`, `max_per_sector`, `stop_loss`, `trailing_stop`, `partial_exit_ratio`, `daily_loss_limit`, `max_hold_days` 확인
- 장운용 시간(`morning_start`, `morning_end`, `afternoon_start`, `afternoon_end`) 확인
- `market_holidays` 최신 반영 확인
- 장전/장중/장후/주말/휴장일 skip 동작 확인
- `preflight` 결과가 실제 실행 차단 기준과 일치하는지 확인

## 6. 브로커-DB 정합성

- `positions`와 실계좌 보유 종목 일치 확인
- `pending_orders` 고립 데이터 없음 확인
- `pending_signals`에 오래된 `processing` / `approved` 항목 없음 확인
- 브로커-DB 정합성 조회 결과 `mismatchCount=0` 확인
- `missingInDb` 자동 복구 가능 여부 확인
- `qtyMismatch` 자동 보정 또는 명확한 운영 절차 존재 확인
- `orphanedDb` 처리 정책 확인
- `/api/positions/reconcile` 수동 실행 결과 확인
- 리컨실 이후 `preflight`가 정상 상태로 복귀하는지 확인
- 주문 직후 / 수동 액션 직후에도 정합성이 유지되는지 확인

## 7. 주문 / 체결 / 포지션 상태

- 주문 전 잔고 / 주문가능금액 / 보유수량 재검증 확인
- 동일 종목 중복 주문 방지 확인
- 수동 주문과 자동 주문 충돌 방지 확인
- 주문 성공 응답 후 체결 여부 후속 확인 로직 점검
- 부분체결 처리 확인
- 미체결 주문 정리 정책 확인
- 지정가 주문 실패 / 시간초과 시 후속 정책 확인
- 주문 실패 원문이 로그에 남는지 확인
- 주문 번호(`orderNo`) 기준 추적 가능 여부 확인
- 포지션 상태가 주문 결과와 일관되게 반영되는지 확인
- 전량청산 / 부분청산 / 재진입 후 포지션 수량과 평단이 맞는지 확인

## 8. 실거래 리스크 제어

- `daily_loss_limit` 정지 동작 확인
- `market_crash_threshold` 정지 동작 확인
- 연속 엔진 오류 자동 정지 기준 확인
- 연속 토큰 오류 자동 정지 기준 확인
- 주문 실패 누적 시 신규 진입 차단 기준 확인
- 시세 / 잔고 이상값 감지 시 fail-safe 정지 확인
- 장 마감 임박 주문 제한 확인
- 유동성 부족 종목 진입 제한 확인
- 동일 섹터 / 동일 종목 집중도 제한 확인
- 재진입 과열 방지 규칙 확인

## 9. 데이터 품질 / 입력 검증

- 시세 응답 필드 누락 시 안전하게 실패하는지 확인
- 잔고 응답 필드 누락 시 안전하게 실패하는지 확인
- 캔들 부족 시 기본값 처리 여부와 영향 확인
- 뉴스 / 스냅샷 / 수급 데이터가 비정상일 때 주문이 과도하게 열리지 않는지 확인
- 이상값(outlier) 방어 확인
- 모의/실전 손익 계산에 수수료/세금 반영 여부 확인

## 10. 관측성 / 알림 / 운영성

- 홈에서 `engineEnabled`, `engineLocked`, `healthStatus`, 최근 이벤트 확인
- `/stats`에서 핵심 운영 지표 확인
- `/api/engine-log`에서 최근 실행 액션 확인
- `engine_state_events`에 설정 변경 / 수동 액션 / 리컨실 이력 적재 확인
- 토큰 오류, 주문 실패, 정합성 오류, 자동 정지 알림 수신 확인
- 같은 오류 반복 시 알림 폭주가 없는지 확인
- 최근 오류 원인을 운영자가 한 화면에서 파악 가능한지 확인
- 실계좌 상태와 UI 상태가 일치하는지 확인

## 11. 전략 / 재진입 / 한도

- 전략 배분 `20 / 40 / 40`이 의도대로 반영되는지 확인
- 종목별 당일 진입 횟수 제한 확인
- `partial_tp` 이후 재진입 경로 확인
- 승인매수와 자동매수의 제한 규칙이 일치하는지 확인
- 전략별 성과를 분리해 볼 수 있는지 확인
- 전략 on/off 또는 위험 전략 격리 수단 존재 확인

## 12. 리허설

- 모의투자 또는 소액 실거래로 최소 1일 이상 리허설
- 수동매수 1건 확인
- 수동매도 1건 확인
- 자동 진입 1건 이상 확인
- 자동 청산 1건 이상 확인
- 리컨실 1건 확인
- 텔레그램 알림 우선순위(`P1/P2/P3`) 수신 확인
- preflight -> 엔진 실행 -> 주문 -> 체결/포지션 반영 -> 로그 적재까지 전체 플로우 확인

## 13. 운영 문서 / 대응 절차

- 엔진 runbook 최신화 확인
- 장애 발생 시 대응 절차 문서 확인
- "이 상태면 거래 중지" 기준 문서화 확인
- "이 상태면 수동 개입" 기준 문서화 확인
- 실전 승격 체크리스트 존재 확인

## 14. 최종 실운용 시작 조건

아래 조건을 모두 만족할 때만 실자금 확대:

- 최근 장중 연속 오류 없음
- 브로커-DB 정합성 불일치 없음
- `preflight`에 `fail` 없음
- 허용되지 않은 `warn` 없음
- 엔진 상태 카드와 실계좌 상태 일치
- 주문 계좌 인증 `pass`
- 토큰 재발급 테스트 성공
- 소액 리허설에서 수동/자동 매매 전부 정상 확인
- 알림 체계 정상 확인
- 운영자 대응 절차 준비 완료

## 15. 우선순위 메모

운영 전 작업 우선순위는 아래 순서를 기본으로 한다.

1. 브로커-DB 정합성 자동복구 완성
2. 토큰 / 인증 / 저장 토큰 경고 정리
3. 주문-체결-포지션 상태 일관성 강화
4. `preflight` 차단 규칙 확정
5. 점검 스크립트 / 테스트 / 린트 전부 통과
6. 리허설 + 알림 + 운영 문서 마무리
