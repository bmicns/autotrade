# Runtime Risks

## 현재 평가

`NEXIO`는 작동하는 시스템이지만, 안정성 관점에서는 몇 가지 운영 리스크가 남아 있다.

## 주요 리스크

### 1. KIS 자격증명 불일치

- 원인:
  과거에는 `KIS_* env`와 `kis_config` DB가 동시에 사용되어 서로 다른 값을 가질 수 있었다.
- 영향:
  `observer`는 동작하지만 `engine`은 토큰 발급 실패하는 식의 분기 장애가 발생할 수 있다.
- 현재 대응:
  런타임은 `kis_config`를 우선 소스로 보고, DB 값이 없거나 실패하면 `env`를 복구 경로로만 사용한다.

### 2. 외부 API 의존성

- 의존 대상:
  `Supabase`, `KIS`, `Telegram`, `DART`
- 영향:
  외부 장애나 네트워크 실패 시 엔진이 조기 종료하거나 일부 단계가 스킵될 수 있다.
- 현재 대응:
  주요 API는 명시적 오류 JSON 또는 로그 기록을 남긴다.

### 3. 장중 검증 부족

- 현재 상태:
  장외 스킵과 observer 수집은 확인했지만, 장중에 실제 스캔/신호/주문/청산 전 과정을 아직 반복 검증하지 않았다.
- 영향:
  실거래 흐름의 마지막 20%가 미검증 상태로 남는다.

### 4. 설정 변경의 이중 저장

- 현재 상태:
  클라이언트 localStorage와 `kis_config` DB가 함께 존재한다.
- 영향:
  브라우저에 오래된 표시값이 잠깐 보일 수 있다.
- 현재 대응:
  hydrate 시 DB 값을 우선 읽어 localStorage를 덮어쓴다.

### 5. Cron / 세션 시간 의존

- 엔진은 `09:30 ~ 15:20 KST` 범위 밖에서 자동 스킵된다.
- GitHub Actions, Vercel Cron, KST 변환이 어긋나면 실행은 성공해도 실질적으로는 장외 스킵만 반복될 수 있다.
- 현재 대응:
  주말, 근로자의 날, 연말 최종 거래일은 코드로 스킵하고, 그 외 공휴일/임시 휴장은 `app_config.market_holidays`로 수동 주입 가능하다.

### 6. 전략 분리 후 배분값 오설정

- 현재 상태:
  `watchlist_pullback`, `surge_momentum`, `institutional_follow` 3개 전략이 각각 별도 배분 비율을 가진다.
- 영향:
  특정 전략 배분이 `0` 또는 과도하게 작으면 신호는 떠도 실매수 수량이 0이 될 수 있다.
- 현재 대응:
  엔진은 배분값을 정규화해서 사용하고, 성과 통계에 `strategyBreakdown`을 노출한다.

## 우선순위

1. 장중 엔진 실검증
2. 주문 후 DB 상태 전이 확인
3. 전략별 배분과 실제 체결 수량 일치 여부 확인
4. 배포 환경과 로컬 `kis_config` 일치 여부 점검
5. observer와 engine 로그 모니터링 정례화
